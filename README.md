> ūüß† Self-curating, navigable, context-budget-aware **agent memory** that judges your session for
> what's worth keeping ‚ÄĒ with no vector database, no embedding pipeline, and no server. Just Markdown
> in a git repo. Works as a Claude Code plugin.

Mnemex (from *Mnemosyne*, the personification of memory, + the engineering suffix *-ex*) is a
specification **and** a Claude Code plugin for capturing the durable knowledge an agent produces ‚ÄĒ
the **what** (domain facts) and the **how** (the patterns and review decisions that govern good
work in a domain) ‚ÄĒ into a plain-Markdown graph that lives in a git repository, organizes itself
into human-like memory tiers (**ūüĒ• hot / ūüĆ§ÔłŹ warm / ‚ĚĄÔłŹ cold**), and forgets what stops being useful while
protecting what is structurally important.

It is designed to give agents long-term, navigable, *context-budget-aware* memory **without a vector
database, without an embedding pipeline, and without a server.** Routing is structural (folder +
index traversal), decay is lazy (computed, never swept), and every mutation is a reviewable git
commit.

## ūüí° Why this exists

Agents are excellent at producing knowledge inside a session and terrible at keeping it. The two
common fixes both have sharp costs:

| ‚ĚĆ Common fix | ūüíł The cost it carries |
|---|---|
| **Stuff everything into context** | Context bloat, cost, and attention dilution. |
| **Embed everything into a vector store** | Infrastructure, an embedding pipeline, an index to operate, and opaque retrieval you cannot read or diff. |

Mnemex takes a **third path**. Knowledge is **files**. Navigation is the **filesystem plus small index
files** you read in chunks. Relevance is a **number you compute on demand** from a usage log, modeled
on the *Ebbinghaus forgetting curve* and *spaced repetition*: things used often stay visible and
cheap to reach; things unused drift down the tiers and, eventually, die ‚ÄĒ unless something still
points at them. The result is a knowledge base that behaves like memory rather than a landfill.

```mermaid

    B -->|"Stuff context"| C[ūüíł Bloat + cost]

    B -->|"Mnemex"| E[ūüďĄ Markdown graph<br/>in git]
    E --> F[ūüĒ• Hot ¬∑ ūüĆ§ÔłŹ Warm ¬∑ ‚ĚĄÔłŹ Cold<br/>self-tiering memory]
```

## ūüß† It remembers like you do ‚ÄĒ automatically

You don't consciously decide to memorize everything that happens in a conversation. Your mind
quietly judges, in the background, what was *significant* and what was noise ‚ÄĒ and keeps the former
without you ever asking it to. **Mnemex is auto-memory in exactly that sense.** You do the work; it
watches the session and decides what is worth keeping.

When a session ends, Mnemex looks at what actually happened and asks the questions a person would:

- **Is this relevant?** ‚ÄĒ does it generalize beyond this one conversation, or was it throwaway scaffolding?
- **Is this significant?** ‚ÄĒ a durable domain fact or a hard-won review decision, versus an incidental detail?
- **Is this novel?** ‚ÄĒ something the graph doesn't already know, versus a restatement of what's there?

It **captures the knowledge that passes** and **ignores the rest.** Concretely, capture extracts
candidate atoms from the session and scores each one `now` / `later` / `not-needed` ‚ÄĒ the explicit
keep / defer / forget judgment. The author doesn't curate, tag, or decide what to file away; that
salience call is Mnemex's job, and what survives later rises or decays on its own through the
ūüĒ• hot / ūüĆ§ÔłŹ warm / ‚ĚĄÔłŹ cold tiers as it gets used or stops being used.

And just like your own memory, being *frequently recalled* is not the same as being *still true*. Mnemex
tracks that too, on a **separate axis**: every fact carries a `verified` clock, and when it hasn't been
re-confirmed within a horizon you set (`freshness_ttl_days`), it's flagged ‚Ź≥ **stale** the next time it's
read ‚ÄĒ even if it's hot ‚ÄĒ so the agent re-checks it against the source instead of confidently repeating
something outdated. Confirm it's unchanged and the clock resets for one cheap stamp; find it's wrong and
the correction flows back in through capture. (Full model:
[`docs/freshness-and-revalidation.md`](docs/freshness-and-revalidation.md).)

The author doesn't worry about any of this. You build; Mnemex remembers what mattered ‚ÄĒ and flags what may
have gone stale. ūüß≠

> The judgment is reviewable, not a black box: capture stages locally and you can inspect or
> un-stage anything (`mnx-status`, `mnx-capture --drop`) before a deliberate `mnx-promote` commits
> it to the shared graph. *Automatic, but never unaccountable.*

## ūü™ô Built to burn fewer tokens

A knowledge base is only worth having if *reading* it is cheap. The two usual approaches quietly tax
every single query: stuffing prior knowledge into the prompt pays for the whole pile on every turn, and
vector-RAG pastes the top-*k* retrieved chunks into context each time you ask. Both get **more** expensive
as the corpus grows.

Mnemex is architected the other way: **tier is literally read cost**, so the tokens a read spends scale
with the *path you take*, not the *size of the graph*. You navigate by reading tiny index heads and open
only the handful of node bodies you actually commit to.

| ūüéĮ Mechanism | ūü™ô Why it spends fewer tokens |
|---|---|
| ūüß≠ **Route, don't retrieve** | You read one-line index heads to *pick a path* (org ‚Üí team ‚Üí cluster). Nothing is pasted into context on spec. |
| ūüĒ• **Hot = top-K, chunk 1** | The routing head is capacity-bounded ‚ÄĒ it stays small even in a huge graph, so the baseline read is bounded regardless of node count. |
| ūüďö **Chunked tier reads, stop early** | Read **Hot** first; it's usually enough. **Warm/Cold** only on demand. You rarely load a whole index, never the whole graph. |
| ūüŹ∑ÔłŹ **Match on denormalized summaries** | Each index row carries the node's `summary`+`aliases`, so you *match without opening a single node body*. |
| ūüĒé **Expand only on commit** | Load only the bodies you'll actually use, within a per-hop token budget ‚ÄĒ beam search, not "load every neighbor". |
| ūüóĎÔłŹ **Self-pruning** | Decay + death keep the routing surface small over time ‚ÄĒ no landfill of dead knowledge to page through. |
| ‚Ź≥ **Trust the fresh, re-check only the stale** | A verified fact is used as-is; you don't re-derive known knowledge from scratch ‚ÄĒ only a *stale* atom triggers a re-check. |

> ūüí° **The payoff:** a read is a few small index-head reads **plus only the node bodies you commit to** ‚ÄĒ
> not the corpus, not a wall of retrieved chunks. Retrieval stays cheap as the graph grows into the
> thousands of nodes, which is exactly where naive context-stuffing and RAG get most expensive.

## ‚öôÔłŹ The four operations

Four of Mnemex's skills are memory operations, each fronted by a slash command. They map to the verbs of
a memory system (two more skills, `mnx-init` and `mnx-status`, handle setup and status ‚ÄĒ see below).
Knowledge writing is split **capture / promote** ‚ÄĒ the `git commit` vs `git push`/PR of memory.

| ūüéõÔłŹ Command | Skill | What it does | Mutates? |
|---|---|---|---|
| ūüĒć `/mnemex:mnx-read` | `mnx-read` | Route ‚Üí read tiered indexes in chunks ‚Üí **overlay** local staged atoms ‚Üí expand only needed nodes ‚Üí **flag stale atoms** for revalidation ‚Üí emit a **usage manifest** ‚Üí append stamps for nodes actually used. | Registry append only (pure w.r.t. knowledge). |
| ‚úćÔłŹ `/mnemex:mnx-capture` | `mnx-capture` | Capture the **current session** (artifact + human review points) ‚Üí extract atoms ‚Üí **score** each `now/later/not-needed` ‚Üí **stage** locally with self-sufficient provenance. Cheap, local, no lock. Also **curates** staging: `--drop <id>` / `--discard-all` un-stage (review via `mnx-status`) ‚ÄĒ the local un-stage and the hard-cap escape valve. | No ‚ÄĒ writes only the local staging tier. |
| ūüöÄ `/mnemex:mnx-promote` | `mnx-promote` | The deliberate merge: flush stamps ‚Üí **reconcile + merge** staged atoms (clean-context sub-agent, HITL on contradictions) ‚Üí **consolidate** the post-merge graph (decay/re-tier/death/edge-hygiene/budget) ‚Üí doctor ‚Üí push ‚Üí clear staging. If a push fails after commit, `--retry-push` lands the existing commit (never re-merges). | Yes ‚ÄĒ gated, atomic, one commit. |
| ūü©ļ `/mnemex:mnx-doctor` | `mnx-doctor` | The validator: checks every invariant (edge targets exist, index matches nodes, denormalized copies are fresh, reverse map consistent, no dangling edges) and can self-heal derived files. | Repair mode only. |

The maintenance pass (`mnx-consolidate`) is **internal** ‚ÄĒ the back half of `mnx-promote`, with no
standalone slash command.

### ūüŹóÔłŹ Bootstrap a whole graph from an existing repo ‚ÄĒ in one command

**You don't start from an empty graph.** Point Mnemex at a repo you already have and it builds the memory
for you:

```bash
/mnemex:mnx-ingest github.com/acme/payments-service     # or a local path
```

`mnx-ingest` reads an entire **code or documentation repository** ‚ÄĒ no live session needed ‚ÄĒ and turns years
of accumulated docs, ADRs, API contracts, and code comments into a live, connected knowledge graph. It does
**not** dump the repo into a vector store. It **distills**: an LLM mines each file for the durable *facts*
and *decisions* worth keeping, **collapses the same fact stated five different ways into one well-sourced
node** (entity resolution), and wires the `[[wiki-links]]` between them ‚ÄĒ so what lands is *distilled
memory*, not a RAG index over your source.

And it stays honest at scale:

- **Two gates, not a thousand.** Approve the scope + routing map once up front, then a single bulk summary at
  the end ‚ÄĒ **never** per-atom review. A monorepo is one decision, not ten thousand.
- **Re-runnable and idempotent.** Run it again after the repo changes and it imports **only the diff**; a
  deleted source file surfaces as an *orphan candidate* for you to judge ‚ÄĒ never silent auto-death.
- **Safe by construction.** Ingest **only stages** (promote stays the sole writer), **never reads secrets**,
  and **never mutates the source**.
- **Higher recall for free.** A shared **gleaning** pass ("what did I miss?") lifts extraction completeness ‚ÄĒ
  and it improves ordinary session capture too.

Everything downstream ‚ÄĒ reconcile, merge, the wiki mesh, consolidate, doctor, push ‚ÄĒ is the **same pipeline**
your daily captures already flow through. Full model: [`docs/corpus-ingestion.md`](docs/corpus-ingestion.md).

Two further skills round out the surface. `/mnemex:mnx-init` is the setup/preflight: it **binds** a

existing one ‚ÄĒ and is what every other command resolves first. For a git-remote graph it runs a

> standard library. If `PyYAML` is missing, the Mnemex commands report it and tell you to install it
> rather than failing cryptically.

A complete walkthrough ‚ÄĒ from install to daily usage, with the hooks that fire automatically ‚ÄĒ is in
[`docs/user-journey.md`](docs/user-journey.md). ūüß≠

### Other agents

Claude Code gets the full plugin experience above (auto-capture hooks, skills). Every other MCP-capable
agent gets the same read/capture/promote loop over the stdio MCP server, wired up with one command.
For step-by-step, per-agent instructions (install ‚Üí what changed ‚Üí how to verify ‚Üí which tier to
expect), see [`docs/agent-setup.md`](docs/agent-setup.md).

```bash
# 60-second start: create a local-folder graph, bind it, and pin it into the entry ‚ÄĒ zero prompts.
uvx openmnemex install --agent <agent> --init-graph --yes

# ‚Ä¶or wire up the server against a graph you already have:
uvx openmnemex install --agent <agent> [--project|--user] [--pin-graph]
```

`--init-graph` is the no-decisions path: it proposes a plain local folder under your mnemex home
(no git remote, no credentials ‚ÄĒ it always succeeds), scaffolds it doctor-clean, binds it as your user
default, and pins that graph into the agent's MCP entry. Nothing to choose, no dead end on the first
read. `--pin-graph` **without** a graph now errors clearly instead of silently writing an unpinned
entry.

| `--agent` | Config written | Tier |
|---|---|---|
| `opencode` | `opencode.json` + `AGENTS.md` block + `.opencode/plugin/mnemex.ts` (auto-capture hook) | Full |
| `gemini-cli` | `.gemini/settings.json` / `~/.gemini/settings.json` + `GEMINI.md` block | Assisted |
| `codex` | `.codex/config.toml` / `~/.codex/config.toml` + `AGENTS.md` block | Assisted |
| `copilot` | VS Code `.vscode/mcp.json` (project scope only ‚ÄĒ no static user-scope file) | Assisted |
| `cursor` | `.cursor/mcp.json` + `.cursor/rules/mnemex.mdc` | Assisted |
| `claude-code` | `.mcp.json` (project) or `claude mcp add` (user) ‚ÄĒ an alternative to the plugin above | Full |

"Assisted" tier means the host has to be told the read/capture/promote judgment procedure via the
instruction file (`AGENTS.md`/`GEMINI.md`/rules block) rather than getting it as a Claude Code skill ‚ÄĒ
see [`LIMITATIONS.md`](LIMITATIONS.md) for exactly what differs. `--pin-graph` bakes the current
directory's resolved graph path into the server config so the agent doesn't need to `cd` there first;
add `--dry-run` to preview the diff or `--check` to verify an existing install; `--uninstall` removes
exactly the Mnemex entry/block and leaves everything else in the file untouched.

> [!NOTE]
> Not on PyPI yet ‚ÄĒ until then, `uvx --from git+https://github.com/kritird/OpenMnemex openmnemex
> install --agent <agent> ...` runs it straight from this repo.

JS-native users can skip `uv`/`pip` entirely once published: `npx openmnemex install --agent
<agent> ...` and `npx openmnemex-mcp` are thin shims that exec the same PyPI package underneath
(`integrations/npm/`, not yet published ‚ÄĒ same not-on-PyPI caveat above applies).
